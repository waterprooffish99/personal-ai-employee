# Data Model: Personal AI Employee System

**Date**: 2026-02-24
**Feature**: 001-ai-employee-system
**Status**: Complete

## Core Entities

### Task
**Description**: Represents a unit of work to be processed by the AI employee
**Fields**:
- `id`: Unique identifier for the task
- `title`: Human-readable title of the task
- `description`: Detailed description of what needs to be done
- `status`: Current state (pending, needs_action, approved, completed, rejected)
- `created_at`: Timestamp when the task was created
- `completed_at`: Timestamp when the task was completed
- `source`: Origin of the task (email, file, etc.)
- `priority`: Priority level (P1, P2, P3, etc.)

### ActionFile
**Description**: Represents an action file created by watchers and processed by Claude
**Fields**:
- `id`: Unique identifier for the action file
- `filename`: Name of the Markdown file
- `path`: Path in the vault where the file is located
- `content`: Raw content of the action file
- `created_at`: Timestamp when the file was created
- `processed_at`: Timestamp when the file was processed
- `status`: Current state (inbox, needs_action, done, pending_approval)

### WatcherEvent
**Description**: Represents an event detected by a watcher component
**Fields**:
- `id`: Unique identifier for the event
- `watcher_type`: Type of watcher that detected the event (gmail, whatsapp, filesystem)
- `source_id`: Identifier from the source system (email ID, message ID, etc.)
- `timestamp`: When the event was detected
- `event_type`: Type of event (new_email, file_change, message_received)
- `event_data`: Raw data from the event source

### ApprovalRequest
**Description**: Represents a request that requires human approval before execution
**Fields**:
- `id`: Unique identifier for the approval request
- `request_type`: Type of action requiring approval (email, payment, etc.)
- `request_data`: Details of the action to be approved
- `created_at`: Timestamp when the request was made
- `approved_at`: Timestamp when the request was approved
- `status`: Current status (pending, approved, rejected)
- `approver`: Identity of the person who approved/rejected

### AuditLog
**Description**: Record of all actions performed by the AI employee for accountability
**Fields**:
- `id`: Unique identifier for the log entry
- `timestamp`: When the action occurred
- `actor`: Who performed the action (AI, human)
- `action_type`: Type of action performed
- `target`: What was affected by the action
- `result`: Outcome of the action (success, failure, pending)
- `approvals`: List of approvals involved in the action
- `dry_run`: Whether this was a dry-run operation

### Plan
**Description**: Structured multi-step plan generated by Claude for complex tasks
**Fields**:
- `id`: Unique identifier for the plan
- `title`: Brief title of the plan
- `description`: Description of what the plan accomplishes
- `steps`: List of ordered steps to execute
- `status`: Current execution status (draft, approved, executing, completed)
- `created_at`: When the plan was created
- `executed_at`: When the plan execution started
- `completed_at`: When the plan execution finished

### CEOBriefing
**Description**: Weekly business intelligence report generated by the AI employee
**Fields**:
- `id`: Unique identifier for the briefing
- `week_ending`: The week this briefing covers
- `revenue_summary`: Summary of revenue for the week
- `bottleneck_analysis`: Identified bottlenecks and issues
- `cost_suggestions`: Suggestions for cost reduction
- `action_items`: Items requiring human attention
- `generated_at`: When the briefing was generated

## State Transitions

### ActionFile States
```
Inbox → Needs_Action → [Pending_Approval → Approved] → Done
```
- Action files start in the /Inbox directory
- Claude moves them to /Needs_Action for processing
- Sensitive actions are moved to /Pending_Approval for human review
- After approval, they move to the /Approved directory
- Completed actions are moved to /Done

### Task States
```
Pending → Needs_Action → Approved → [Executing] → Completed | Rejected
```
- Tasks start as pending when detected by watchers
- Claude sets them to needs_action when it begins processing
- Human approval changes status to approved
- Execution may involve multiple phases before completion
- Tasks can be rejected at any approval point

## Relationships

- **WatcherEvent → ActionFile**: One-to-many (one event can create multiple action files)
- **ActionFile → Task**: One-to-one (each action file represents one task)
- **Task → ApprovalRequest**: One-to-many (one task may require multiple approvals)
- **Task → Plan**: One-to-one (complex tasks may require a plan)
- **Plan → Task**: One-to-many (one plan may create multiple subtasks)
- **AuditLog → (Task|ActionFile|ApprovalRequest)**: One-to-one (each log entry links to one entity)
